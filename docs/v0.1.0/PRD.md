# 女友（Girlfriends）v0.1.0 产品需求文档

## 1. 文档概览

### 1.1 文档目的

本文档用于定义《女友（Girlfriends）》v0.1.0 的产品目标、系统边界、角色内容、多人规则与验收标准。

本文档只描述产品需求与玩家体验，不展开技术实现方案。

### 1.2 版本定位

v0.1.0 定位为完整第一版，而非仅验证概念的原型版。

本版本需要提供完整可游玩的角色养成、委托推进、跟随祝福、家园同居与多人竞争体验。

本版本的内容核心是五位首发角色的完整主线与长期陪伴循环。

### 1.3 产品一句话定义

《女友（Girlfriends）》是一款面向 Minecraft 生存模式的全年龄功能性美少女养成模组，通过互动、协作、养成与轻剧情推进，为玩家提供可长期陪伴并反哺原版生存、建设与战斗循环的角色体验。

### 1.4 名称与关系表达边界

“女友（Girlfriends）”保留双语义，既可以被理解为恋爱关系中的女友，也可以被理解为亲密的女性伙伴或闺蜜。

本模组整体表达偏向羁绊、陪伴、约定与家园伙伴，不涉及成人向内容。

五位角色设定上均为成年女性，但产品文案不反复强调年龄。

## 2. 产品背景与目标

### 2.1 产品背景

Minecraft 原版提供了强自由度的探索、采集、建设与战斗体验，但缺少可长期成长、具备明确情感连接和玩法反哺的陪伴型角色系统。

本模组希望在不颠覆原版流程的前提下，补充“探索发现角色—建立关系—完成愿望—获得陪伴收益—形成长期生活感”的内容层。

### 2.2 产品目标

1. 为玩家提供五位具备鲜明主题、独立愿望与差异化玩法价值的可攻略角色
2. 形成以好感度、委托、跟随祝福、家园同居为核心的长期养成闭环
3. 让角色成为原版生存、建设、战斗与探索循环的增强者，而不是替代者
4. 在多人游戏中提供可竞争、可转移、可协助但边界清晰的攻略关系规则
5. 将五位首发角色的固定委托线打造为稳定核心内容，后续版本主要通过增加新角色扩展玩法

### 2.3 非目标

1. 不做成人向、露骨关系表达或相关暗示内容
2. 不通过大段文本对话推进剧情，剧情表达以短句任务卡、行为反馈、少量聊天栏文本与环境变化为主
3. 不彻底颠覆原版主线、生存、建设与战斗节奏
4. 不在 v0.1.0 中实现角色召回、主动寻人线索系统与大型图鉴剧情回顾系统
5. 不在 v0.1.0 中修改五位首发角色固定委托线的整体主线方向

### 2.4 核心设计原则

1. 陪伴增强原版，而非替玩家玩游戏
2. 角色收益可感知，但不过度自动化
3. 剧情表达轻量化、事件化、环境化
4. 多人规则可竞争，但需要保证可理解与可预期
5. 首发角色主线稳定，后续扩展以新增角色为主

## 3. 目标用户与核心体验

### 3.1 目标用户

1. 喜欢 Minecraft 生存、建设与中长期养成目标的玩家
2. 喜欢角色陪伴、轻剧情推进与收集成长体验的玩家
3. 喜欢 Galgame 氛围，但不希望被大段文本打断游玩节奏的玩家
4. 喜欢在多人服务器中形成轻竞争与协作互动的玩家

### 3.2 核心体验目标

1. 玩家能够在探索世界时自然遇见角色，并对她们的栖居环境、行为与个性产生第一印象
2. 玩家能够通过委托、赠礼与陪伴逐步解锁更高层次的互动和更明确的玩法收益
3. 玩家能够明显感知每位角色在生存、建设、战斗、航海或探索上的差异化价值
4. 玩家在完成固定委托后，能够感受到角色愿望被逐步实现，以及自己与她的关系发生变化
5. 玩家在后期仍有理由持续与角色互动，而不是在主线结束后失去目标

## 4. 核心玩法闭环

### 4.1 总体闭环

- 玩家通过探索发现五位角色及其临时庇护所
- 玩家通过赠礼、互动与完成委托提升好感度
- 好感度提升后，玩家逐步解锁新的固定委托、随机委托、跟随权限与亲密关系确认内容
- 角色在跟随玩家时提供差异化祝福与少量主题行为，反哺原版循环
- 玩家继续推进委托线，完成角色愿望并获得终章专属功能奖励
- 当关系达到家园同居阶段后，玩家可以邀请其中一位角色入住家园，形成长期陪伴与稳定收益
- 固定委托全部完成后，角色继续以随机委托、礼物、跟随与家园互动构成长线循环

### 4.2 四大核心系统

1. 好感度系统负责承载关系成长、阶段解锁与多人绑定资格
2. 委托系统负责提供主要可玩内容与角色主线推进
3. 协助与祝福系统负责把角色价值转化为原版玩法收益
4. 家园系统负责承接后期关系沉淀与长期生活感

## 6. 系统需求

### 6.1 角色发现与世界存在

1. 每位角色在同一世界中只存在一个本体角色
2. 每位角色的临时庇护所可按原版结构生成逻辑生成多个
3. 本期不提供主动定位或寻人线索系统，玩家需要通过探索目标群系或维度自然发现角色
4. 临时庇护所为轻量主题结构，承担首次发现、重生承载、主题展示与部分委托目标功能
5. 晚萤的庇护所仅生成于下界，幽若的庇护所仅生成于末地，其余三位角色的庇护所仅生成于主世界

### 6.2 好感度系统

#### 6.2.1 基本规则

1. 每位玩家对每位角色独立计算好感度
2. 内部采用 0 到 1000 的数值制
3. 玩家不可见具体数值，只能通过 UI 中的模糊进度条感知关系进度
4. 好感度主要通过委托、赠礼、陪伴与家园相处增长
5. 极少数负面事件会导致好感下降或清零

#### 6.2.2 阶段划分

| 阶段 | 数值区间 | 含义 | 主要解锁内容 |
| --- | --- | --- | --- |
| 陌生 | 0~100  | 初见与试探期 | 基础互动、少量低阶随机委托 |
| 熟悉 | 100~200  | 建立初步信任 | 第 1 条固定委托、更多赠礼反馈 |
| 信赖 | 200~500  | 形成稳定合作 | 第 2 到 4 条固定委托、更多随机委托模板 |
| 爱慕 | 500~700  | 形成明显偏向与绑定资格 | 第 5 到 7 条固定委托、多人绑定竞争规则开启 |
| 亲密 | 700~900  | 正式确认亲密关系 | 第 8 到 9 条固定委托、跟随与祝福 |
| 家园同居 | 900~1000  | 进入稳定生活关系阶段 | 第 10 条固定委托、家园入住资格 |

#### 6.2.3 下降规则

1. 厌恶礼物会导致好感下降
2. 玩家攻击或误伤角色会导致好感下降
3. 家园目击争执事件会导致少量好感下降
4. 角色死亡会导致该角色对所有玩家的好感、绑定、委托与家园状态完全清零
5. 不设置长期不互动导致的自然衰减

### 6.3 委托系统

#### 6.3.1 总体规则

1. 委托分为随机委托与固定委托两类
2. 随机委托不承担主线叙事，负责日常循环与小额收益
3. 固定委托承担角色故事推进、愿望实现、亲密确认与终章奖励发放
4. 角色头顶需要使用两种不同图标区分随机委托与固定委托

#### 6.3.2 固定委托规则

1. 每位角色拥有 10 条固定委托
2. 分布为熟悉 1 条、信赖 3 条、爱慕 3 条、亲密 2 条、家园前置 1 条
3. 当玩家好感度达到 700 后，不自动进入亲密阶段，而是需要通过专属固定委托或关键交互显式确认
4. 固定委托接取后一直保留，直到完成为止，不会失败、不会过期
5. 同一条固定委托只允许一名玩家接取并完成，但其他玩家可以自由协助
6. 奖励、推进资格与关系提升归属于接取者

#### 6.3.3 随机委托规则

1. 每位角色拥有 8 类随机委托模板
2. 随机委托按好感阶段逐步解锁更高阶类型与奖励
3. 随机委托在 5 到 10 个游戏日之间随机刷新
4. 若随机委托在下次刷新前未完成，则自动过期且无惩罚
5. 随机委托主要奖励小额好感度与主题资源，不投放破坏原版节奏的稀有装备

#### 6.3.4 委托槽规则

1. 每个角色同一时间最多只发布 1 个委托
2. 该唯一委托槽中的内容可以是随机委托，也可以是固定委托，但二者互斥
3. 只要任意玩家仍持有该角色未完成的委托，她就不会发布新的委托
4. 玩家可以同时持有多位不同角色的委托，但同一角色的委托仍受其单槽规则限制

### 6.4 礼物系统

#### 6.4.1 偏好档位

1. 每位角色具有最喜好、喜好、普通接受与厌恶四档礼物反馈
2. 每位角色默认仅设置 1 种最喜好物品
3. 产品能力上支持后续附属内容为角色增加更多最喜好物品
4. 礼物偏好采用渐进揭示，不在初始状态完全公开
5. 玩家通过赠礼反馈、固定委托推进与界面记录逐步掌握偏好信息

#### 6.4.2 每日收益规则

1. 每位玩家对每位角色的礼物收益按日独立计算
2. 每日通过礼物获得的好感度最多增长 15 点
3. 角色在当日礼物收益耗尽后，会拒绝继续收礼，并以聊天栏文本提示“你今天已经送我很多礼物了”这类反馈
4. 基础档位固定如下表

| 档位 | 基础好感变化 |
| --- | -- |
| 最喜好 | +5 |
| 喜好 | +4 |
| 普通接受 | +2 |
| 厌恶 | -2 |

#### 6.4.3 赠礼公式

单次赠礼获得的实际好感变化值按下列方式计算

`礼物本身应提供的好感度增长量 × √((16 - 当日已增加的好感度) / 16)` 

该公式同时作用于最喜好、喜好、普通接受与厌恶礼物。

当当日可增长空间越少时，礼物收益越接近于零，从而形成明确的边际收益递减。

### 6.5 跟随与祝福系统

#### 6.5.1 跟随定位

1. 跟随系统以轻跟随为核心定位
2. 角色跟随时以陪伴、表现与祝福反哺为主，不承担强自动化采集或代打职责
3. 晚萤是唯一明显偏战斗支援的角色

#### 6.5.2 固定祝福效果

| 角色 | 祝福效果 |
| --- | --- |
| 渔溪 | 玩家划船速度提高 50%，钓鱼时有 25% 概率获得双倍收益 |
| 梅疏 | 玩家挖掘矿物时有 25% 概率额外掉落对应矿物 |
| 沫沫 | 玩家不会吸引蜘蛛、熊猫、狼、北极熊、羊驼、山羊、蜜蜂与疣猪兽的仇恨 |
| 晚萤 | 玩家近战伤害增加 40%，受到的伤害降低 25% |
| 幽若 | 玩家投出的末影珍珠有 25% 概率不消耗 |

#### 6.5.3 跟随行为边界

1. 梅疏、沫沫、幽若在玩家位于自己周围 5 格内、且目标方块位于自己周围 2 格内时，可以尝试进行极短距离采集
2. 晚萤在跟随时会主动帮助攻击索敌玩家的敌对生物
3. 其他角色不提供明显自动战斗或远距离自动采集能力

### 6.6 家园系统

#### 6.6.1 入住规则

1. 当玩家与角色好感度达到 900 以上，并完成对应家园前置固定委托后，可邀请其入住家园
2. 家园入住需要通过双人床建立绑定
3. 每位玩家同一时间仅可邀请 1 位家园伙伴入住
4. 玩家可以与多名角色保持亲密关系，但不能同时让多名角色作为家园伙伴入住
5. 更换家园伙伴前，玩家必须先显式解除当前家园同居绑定

#### 6.6.2 家园收益

1. 当玩家与家园伙伴同时处于绑定双人床 20 格内时，玩家每 10 秒恢复 1 点生命值
2. 当玩家与家园伙伴同时处于绑定双人床 20 格内时，每游戏日增加 2 点好感度
3. 上述家园好感增长每日最多触发 1 次

#### 6.6.3 家园冲突事件

1. 当玩家已与多名角色建立亲密关系，并将其中一位带回家园，被当前家园伙伴目击时，会触发争执与吃醋事件
2. 事件通过聊天栏文本表达角色情绪，不进入长对话模式
3. 事件会使当前家园伙伴与来访角色都对该玩家随机扣除 3 到 5 点好感度
4. 该事件对每位玩家每个游戏日最多触发 1 次

### 6.7 多人绑定与竞争规则

1. 角色在初始状态下不绑定任何玩家
2. 每位玩家对角色的好感度独立计算
3. 当某位玩家对某角色达到爱慕阶段后，该角色不再向其他玩家发布固定委托，也不再接受其他玩家礼物
4. 其他玩家对该角色的好感度仅能通过很久才刷新的委托继续缓慢增长
5. 若其他玩家的好感度超过当前绑定玩家，则进入 3 个游戏日的动摇期，并向原绑定玩家发出预警
6. 宽限期结束后，若后来者仍领先，则绑定转移到新的领先玩家
7. 绑定转移只影响后续固定委托接取资格，不清空玩家既有的好感度或已完成固定委托记录
8. 当某位玩家与角色正式确立亲密关系后，该角色也不再向其他玩家发布随机委托，其他玩家将无法继续攻略该角色
9. 固定委托与随机委托均采用单人接取、单人完成、他人可自由协助、奖励仅归接取者的规则
10. 终章专属功能奖励按玩家唯一原则发放，即每位玩家对每位角色仅首次通关时可领取一次功能奖励

### 6.8 角色死亡与重生规则

1. 角色死亡后会失去全部物品与记忆
2. 角色死亡后，对所有玩家的好感度、固定委托进度、随机委托进度、绑定关系与家园状态全部清零
3. 玩家已经取得的终章奖励相关物品、方块、配方或其他玩家资产保留，不因角色死亡而回收
4. 角色重生位置应为离死亡位置最近、且已经生成的对应角色临时庇护所
5. 若当时不存在符合条件的已生成庇护所，则角色暂不重生，待后续探索生成对应结构后再出现
6. 晚萤只能在下界庇护所重生，幽若只能在末地庇护所重生，沫沫、渔溪、梅疏只能在主世界庇护所重生
7. 角色本体存在状态、庇护所发现与生成后的可用状态、死亡后的待重生状态，以及玩家与角色之间的关系状态，均属于必须跨世界长期保留的核心产品状态

### 6.9 交互界面与表现需求

1. v0.1.0 需要提供角色交互界面
2. 交互界面至少包含模糊好感进度条、当前委托状态、赠礼入口、跟随与停留操作、家园邀请操作与已知偏好记录
3. 角色头顶需要明确展示随机委托图标与固定委托图标
4. 关键关系变化、礼物拒绝、家园争执等情境允许使用少量聊天栏文字强化反馈

## 7. 角色共通设计模板

### 7.1 固定委托结构模板

1. 第 1 条固定委托用于建立角色主线方向与初步信任
2. 第 2 到 4 条固定委托用于深化角色习惯、能力与愿望基础
3. 第 5 到 7 条固定委托用于推动角色真正面对自己的长期愿望
4. 第 8 条固定委托用于显式确认亲密关系

#### 8.4.6 少量成长反馈

3. 她在庇护所周边的巡逻与警戒表现会更明显

#### 8.4.7 终章奖励方向

晚萤的终章奖励方向为战斗与防护类长期功能奖励。

该奖励应强化近战、承伤或战备效率，但不直接让玩家跳过原版战斗过程。

### 8.5 幽若（youruo）

#### 8.5.1 角色定位

幽若享受研究带来的快乐，经常采集紫颂果并摆弄仪器。

她希望提取出能够让人自由瞬移的物质，把末地的诡秘规律变成真正可被理解与应用的成果。

她的玩法价值主要体现在末影探索、研究推进、空间感与瞬移主题。

#### 8.5.2 生成与日常行为

幽若生成于末地。

她的临时庇护所应体现隔离、紫色研究氛围、样本摆放与简易实验空间。

她会收集紫颂果、观察末影相关现象，并将样本带回屋内研究。

#### 8.5.3 礼物偏好方向

- 最喜好物品为末影珍珠
- 喜好方向为紫颂果、末地样本、研究材料与空间感强的物件
- 普通接受方向为玻璃容器、基础记录与常规补给资源
- 厌恶方向为低价值腐坏物与对研究毫无帮助的混乱礼物

#### 8.5.6 少量成长反馈

1. 庇护所中研究台、瓶器、样本与草图逐步增加
2. 幽若会随身携带更多样本或记录物
3. 她在屋内摆弄仪器与观察样本的行为会更频繁

#### 8.5.7 终章奖励方向

幽若的终章奖励方向为瞬移与末影类长期功能奖励。

该奖励应强化末影珍珠、空间移动或末地探索收益，但不能让玩家彻底跳过原版风险与路径判断。

## 9. 终章后长期循环

1. 固定委托全部完成后，角色继续按当前关系阶段发布随机委托
2. 终章通关后，玩家仍可通过礼物、陪伴、跟随与家园互动持续维持关系体验
3. 终章后不再重复发放同一玩家对同一角色的功能型终章奖励
4. 终章后仍允许角色在多人场景中继续作为被绑定、跟随、家园或陪伴对象存在

### 10.2 软性平衡原则

1. 随机委托奖励应始终以小额好感度与主题资源为主
2. 固定委托奖励可以更有阶段感，但不能直接破坏原版资源递进
3. 终章奖励应体现强主题价值，但仍需保留玩家亲自探索、采集、建造或战斗的必要性

## 11. 未来规划候选项

1. 角色召回功能可作为后续版本候选机制
2. 英文显示名暂未确定，后续由本地化阶段统一确认
3. 如有需要，可另行编写独立游戏设计文档，对数值、叙事表现与交互节奏做进一步细化
4. 后续版本可以继续补充每位角色的随机委托模板数量
5. 后续版本可以通过新增可攻略角色扩充整体内容深度

## 12. 验收标准

### 12.1 版本验收项

1. 五位角色均可在各自指定环境中自然发现，且每位角色的本体在同一世界中唯一
2. 五位角色均具备完整的 10 条固定委托设计与 8 类随机委托模板
3. 好感度系统、礼物系统、跟随祝福系统、家园系统与多人绑定规则均有清晰闭环
4. 五位角色均具备明确主题、差异化收益与终章奖励方向
5. 文档已明确全年龄边界、不做成人向、不做长文本对话与不颠覆原版流程的高优先级约束
6. 终章后长期循环已定义，不会出现角色主线完成后内容完全中断的情况

### 12.2 体验验收项

1. 玩家能够在不阅读大段文本的情况下理解角色愿望推进
2. 玩家能够清楚分辨随机委托与固定委托的定位差异
3. 玩家能够感知不同角色在原版循环中的实际功能价值差异
4. 多人玩家能够理解绑定、动摇期、转移、亲密锁定与家园争执等规则
5. 家园同居与终章奖励能为后期继续游玩提供明确动力
